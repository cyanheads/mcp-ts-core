# 0.9.2 — 2026-05-22

Breaking: `zod` moves to `peerDependencies` ([#143](https://github.com/cyanheads/mcp-ts-core/issues/143)) — existing consumers must `bun add zod@^4.4.3` to upgrade. Also lands two long-standing `field-test` helper fixes (zsh `status` clash, concurrent-session collisions) and a `multi-server-orchestration` skill + `list-skills` script for sub-agent fanouts.

## Added

- **`multi-server-orchestration` skill** — workflow for parallel sub-agent fanouts across MCP server projects (greenfield, maintenance, release passes). Per-scenario references at `skills/multi-server-orchestration/references/`. Sub-agents spawned via the Agent tool don't inherit the parent's skill registry, so the skill mandates an orient block in every sub-agent prompt that runs the new `list-skills` script.
- **`scripts/list-skills.ts`** — parses YAML frontmatter from project-local `SKILL.md` files and prints an index. Wired into `package.json` `files:` and exposed as `bun run list-skills` in `templates/`.
- **`audit:refresh` npm script** — `rm -f bun.lock && bun install && bun audit`, added to root `package.json` and `templates/package.json`. Use when `bun audit` (via `devcheck`) flags a transitive advisory: Bun's `bun update` is sticky on transitive resolutions, so a stale-lockfile false positive can disguise an already-patched dep. If the advisory survives the refresh, it's real.

## Changed

- **`zod` promoted `dependencies` → `peerDependencies`** ([#143](https://github.com/cyanheads/mcp-ts-core/issues/143)) — required, not optional (the framework re-exports `z` from the main entry). Fixes TS2883 in consumers building with `declaration: true` (the framework default) without a top-level `zod` — the emitted `.d.ts` referenced `ZodObject` from the framework's nested `node_modules/zod`, which the consumer's `tsc` couldn't name portably. **Migration:** `bun add zod@^4.4.3` (or a matching range); `bunx @cyanheads/mcp-ts-core init` scaffolds it automatically for new projects.
- **`format-parity` lint walks each union branch.** Previously sampled only the first branch of `z.union` / `z.discriminatedUnion`, so a field missing from a later list-vs-detail formatter branch slipped through. The walker now produces one synthetic sample per branch, runs `format()` against each, and dedupes diagnostics by output path. `format-parity-walk-failed` and `format-parity-threw` warnings stay total — the first variant that throws short-circuits the variant loop.
- **Fuzz generators recognize Zod 4 string-format subclasses.** `z.email()`, `z.url()`, and `z.uuid()` in Zod 4 are no longer `instanceof ZodString`. `zodToArbitrary` falls back to a string-typed arbitrary when `_def.type === 'string'`, so fuzz suites stop generating raw integers/booleans against URL- or email-shaped inputs. `schema-format-portability` still flags `z.url()` by default — the linter and the fuzz generator answer different questions.
- **HTTP landing page refactor** — pulls toward [#94](https://github.com/cyanheads/mcp-ts-core/issues/94) (not closing). Single ambient accent wash (was two radials), mono lowercase section heads with thin accent bar, card radius shrunk, padding tightened, mutability spine `3px → 2px`, no hover lift, no footer divider, description clamp `3 → 2` lines. Tool-card pills split into mutability (meta strip alongside scope chips) and auxiliary (open-world/task/app, beside the title). Source links moved off the header — tool: actions row; resource/prompt: optional footer.
- **`fast-xml-parser` unpinned to `^5.8.0`** across `peerDependencies` and `devDependencies`; `resolutions` entry dropped; `devcheck.config.json` `outdated.allowlist` cleared. Framework only uses `XMLParser` (never `XMLValidator`) with `processEntities: false, htmlEntities: false` — neither 5.8.0's `XMLValidator` sibling-package extraction notice nor tightened DOCTYPE entity validation can reach our call sites. Reverses the defensive `0.9.1` pin. Refs [cyanheads/pubmed-mcp-server#55](https://github.com/cyanheads/pubmed-mcp-server/issues/55).

## Fixed

- **`field-test` zsh `local status` clash** ([#117](https://github.com/cyanheads/mcp-ts-core/issues/117)) — `mcp_init` and `mcp_call` declared `local status`, colliding with zsh's read-only `$status` builtin (alias for `$?`) and breaking the helper on default macOS via the documented `. /tmp/mcp-field-test.sh` sourcing pattern. Renamed to `local code`.
- **`field-test` concurrent same-project sessions collide** ([#144](https://github.com/cyanheads/mcp-ts-core/issues/144), generalizes [#90](https://github.com/cyanheads/mcp-ts-core/issues/90)) — every per-session path (`PREFIX`, `STATE_FILE`, `BUILD_LOG`, `SERVER_LOG`, and the helper script itself at `/tmp/mcp-field-test.sh`) was derived from `$PWD` or hardcoded, so two agents in one project tree raced on all of them. Fix: (1) the agent generates a 10-char alphanumeric ID and writes the helper to `/tmp/<project>-field-test-<ID>.sh`, sourcing that exact path on every later call; (2) the helper itself is now **stateless** — every function takes its IDs as positional args (`mcp_init <url>`, `mcp_call <url> <sid> <method> [params]`, `mcp_log <log>`, `mcp_stop <pid> <log>`), `mcp_start` prints `ready pid=<n> url=<u> port=<n> log=<path>`, and the agent threads them through. Server logs and call scratch use `mktemp` for per-process uniqueness.
- **`devcheck` strips cosmetic `Registry URL must be` noise** emitted by Bun's node-shim when depcheck encounters `cloudflare:*` virtual-module specifiers in Workers tests. The diagnostic never affected the depcheck report — `devcheck.config.json` already lists `cloudflare:*` in `depcheck.ignores` — but the stderr leaked into the run summary.
